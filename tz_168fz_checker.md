# ТЗ: Сервис проверки текстов email-рассылок на соответствие 168-ФЗ

**Версия:** 1.1  
**Дата:** март 2026  
**Статус:** Готово к разработке  

---

## 1. Контекст и цель

### 1.1. Проблема

С 1 марта 2026 года вступил в силу Федеральный закон № 168-ФЗ от 24.06.2025, добавивший статью 10.1 в Закон «О защите прав потребителей». Он обязывает бизнес использовать русский язык в B2C-коммуникациях. Параллельно рекламные email-рассылки регулируются ст. 5 ч. 5 п. 1 и ч. 11 ФЗ-38 «О рекламе», отсылающей к требованиям ФЗ-53 «О государственном языке».

Маркетологи не имеют удобного инструмента для проверки текстов перед отправкой. Ручная проверка по нормативным словарям (ruslang.ru) занимает часы и требует юридических знаний.

### 1.2. Цель сервиса

Веб-приложение, которое принимает текст email-рассылки и за секунды выдаёт:
- список слов и фраз с нарушениями
- классификацию каждого нарушения по типу и норме закона
- готовые замены для каждого проблемного слова
- итоговый «чистый» вариант текста

### 1.3. Целевая аудитория

- Маркетологи и email-специалисты без юридического образования
- Редакторы, копирайтеры в e-commerce
- Контент-менеджеры CRM/ESP-платформ (возможная интеграция)

## 2. Нормативная база (обязателен точный учёт)

Агент должен реализовать логику строго в соответствии со следующими нормами:

| Норма | Реквизиты | Суть |
|---|---|---|
| Критерий допустимости иностранного слова | Ч. 6 ст. 3 ФЗ-53 (ред. ФЗ-52 от 28.02.2023) | Слово допустимо только если: (1) нет общеупотребительного рус. аналога И (2) есть в нормативных словарях |
| Нормативные словари | Распоряжение Правительства № 1102-р от 30.04.2025 | 4 словаря: Орфографический ИРЯ РАН, Орфоэпический ИРЯ РАН, Словарь иностр. слов ИЛИ РАН, Толковый СПбГУ |
| Требования к потребит. информации | Ст. 10.1 ЗоЗПП (введена 168-ФЗ от 24.06.2025) | Информация для потребителей — на русском языке |
| Запрет в рекламе | Ч. 5 ст. 5 п. 1 ФЗ-38 | Нельзя использовать иностранные слова, искажающие смысл |
| Отсылка к языковому законодательству | Ч. 11 ст. 5 ФЗ-38 | Реклама обязана соблюдать требования о государственном языке |
| Равнозначность оформления | Ч. 2 ст. 3 ФЗ-53 (ред. ФЗ-52 от 28.02.2023) | Цвет, тип, размер шрифта — идентичны для рус. и иностр. текста |
| Исключение: товарные знаки | П. 4 ст. 10.1 ЗоЗПП; ч. 3 ст. 3 ФЗ-53 | ТЗ, фирм. наименования — не регулируются |
| Исключение: техрегламенты ЕАЭС | П. 4 ст. 10.1 ЗоЗПП | USB, LED, NFC, ISO, QR и др. — допустимы |
| Ответственность | Ч. 1 ст. 14.8 КоАП (5–10 тыс. ₽ для ЮЛ); ч. 1 ст. 14.3 КоАП (100–500 тыс. ₽ для ЮЛ) | Штрафы за каждое нарушение отдельно |

## 3. Функциональные требования

### 3.1. Ввод текста

Два равнозначных способа ввода основного текста — пользователь использует один из них:

- **Вставка текста** — вставить plain text или HTML-код письма в textarea напрямую
- **Загрузка файла** — загрузить файл `.txt` или `.html`

В обоих случаях сервис анализирует только видимый текст: HTML-теги, атрибуты и системные строки из анализа исключаются (подробнее — п. 4.7).

**Текст с изображений (MVP)** — отдельное дополнительное поле textarea «Текст с баннеров и изображений в письме». Пользователь вручную копирует туда текст, видимый на картинках. Это поле проверяется тем же алгоритмом, результаты объединяются с основными. Автоматическое распознавание текста с изображений — в v1.1 (Tesseract.js) и v2.0 (OpenRouter Vision).

Максимальный объём: 50 000 символов за один запрос (суммарно по обоим полям).

### 3.2. Алгоритм проверки (реализовать последовательно для каждого слова)

**Предварительный шаг — фразовый матчинг (longest match).**
Перед пословным разбором пройти по тексту и найти все многословные фразы из `glossary.json` (например, `flash sale`, `new collection`, `add to cart`). Использовать принцип longest match: если `add to cart` найден как фраза, отдельно `add`, `to`, `cart` не проверяются. Найденные фразы помечаются целиком как одно нарушение. Оставшиеся слова разбираются пословно по шагам ниже.

```
ДЛЯ КАЖДОГО слова (или фразы после longest match) в тексте:

Шаг 1. Написание кириллицей?
  НЕТ → перейти к Шагу 4 (латиница)
  ДА  → перейти к Шагу 2

Шаг 2. Является ли товарным знаком из белого списка?
  ДА  → ДОПУСТИМО (исключение: п. 4 ст. 10.1 ЗоЗПП)
  НЕТ → Шаг 3

Шаг 3. Есть ли в parsed_dictionary.json (извлечён из PDF словарей, см. п. 4.5)?
  ДА  → ДОПУСТИМО (ч. 6 ст. 3 ФЗ-53 + Распоряжение № 1102-р)
  НЕТ → НАРУШЕНИЕ (тип: "кириллица — не в словарях")

Шаг 4 (латиница). Является ли товарным знаком из белого списка?
  ДА  → ДОПУСТИМО
  НЕТ → Шаг 5

Шаг 5. Содержится ли в whitelist технических аббревиатур (tech_abbrev.json)?
  ДА  → СЕРАЯ ЗОНА (тип: "TECH_ABBREV" — см. п. 3.3)
  НЕТ → НАРУШЕНИЕ (тип: "латиница — не разрешена")
```

### 3.3. Классификация нарушений

Каждое найденное нарушение получает один из трёх типов:

| Код | Тип | Описание | Применимая норма |
|---|---|---|---|
| `LAT_PROHIBITED` | Латиница — запрещена | Слово на латинице, не ТЗ и не техаббревиатура | Ч. 2 ст. 10.1 ЗоЗПП; ч. 6 ст. 3 ФЗ-53 |
| `CYR_NOT_IN_DICT` | Кириллица — не в словарях | Заимствование кириллицей, отсутствующее в 4 словарях | Ч. 6 ст. 3 ФЗ-53; Распоряжение № 1102-р |
| `TECH_ABBREV` | Техническая аббревиатура — спорно | Wi-Fi, USB, LED и др.: формального исключения для рекламных текстов нет, практика не сложилась | П. 4 ст. 10.1 ЗоЗПП (косвенно); дело № А06-262/2021 — в пользу перевода |

### 3.4. Вывод результатов

**Верхняя плашка — итог проверки:**
- `✓ Нарушений не найдено` — зелёная
- `⚠ Найдено N спорных аббревиатур` — жёлтая, если все нарушения только `TECH_ABBREV`
- `✗ Найдено N нарушений` — красная, если есть хотя бы одно `LAT_PROHIBITED` или `CYR_NOT_IN_DICT`

**Список нарушений** — каждая запись содержит:
- Само слово/фраза (выделено в тексте)
- Тип нарушения (код + читаемое описание)
- Ссылка на норму закона (кликабельная, со стр./номером статьи)
- Предлагаемые замены (1–3 варианта, от наиболее до наименее официального)
- Уровень риска: `ВЫСОКИЙ` (LAT_PROHIBITED) / `СРЕДНИЙ` (CYR_NOT_IN_DICT) / `НИЗКИЙ` (TECH_ABBREV)

Для `TECH_ABBREV` отображать отдельную плашку-пояснение:
> «Технические аббревиатуры (Wi-Fi, USB, LED и др.) формально не выведены в исключения для рекламных текстов. Практика применения 168-ФЗ ещё не сложилась. Суд в деле № А06-262/2021 занял позицию в пользу перевода. Риск низкий, но не нулевой — решение на ваше усмотрение.»

**Исправленный текст:**
- Кнопка «Применить все замены» — подставляет первый вариант замены для каждого нарушения
- Кнопка «Копировать исправленный текст»
- Diff-режим: оригинал и правка рядом с подсветкой изменений

### 3.5. Управляемые списки пользователя

Сервис поддерживает два редактируемых списка (внутренняя база нормативных слов не ведётся — она заменена парсингом PDF, см. п. 4.5):

**а) Белый список товарных знаков** — пополняется пользователем  
Структура:
```json
{
  "name": "Wildberries",
  "type": "trademark",
  "registration": "Роспатент",
  "note": "Зарег. ТЗ — исключение по п. 4 ст. 10.1 ЗоЗПП"
}
```

**Правила совпадения для товарных знаков:**
- Сравнение **case-insensitive**: `Wildberries`, `WILDBERRIES`, `wildberries` — одно и то же
- **Без морфологии**: только точное совпадение токена с именем ТЗ, без учёта падежей и суффиксов
- **Совпадение внутри составного токена**: если ТЗ входит в составное слово через дефис (например `Wildberries-доставка`), весь токен считается допустимым
- **Многословные ТЗ** (например `Fix Price`) матчатся как фраза, аналогично логике longest match из п. 3.2

**б) Корпоративный глоссарий замен** — пользователь задаёт предпочтительные замены  
Структура:
```json
{
  "original": "sale",
  "replacements": ["распродажа", "сейл"],
  "preferred": "распродажа"
}
```

## 4. Технические требования

### 4.1. Архитектура

Сервис реализуется как **статическое фронтенд-приложение** — набор HTML/CSS/JS файлов без серверной части. Вся логика проверки выполняется в браузере пользователя.

Размещение для доступа команды: **GitHub Pages**. Репозиторий публичный или приватный с включённым GitHub Pages. Деплой — через GitHub Actions при каждом пуше в `main`. Никакого бэкенда, базы данных и Docker не требуется.

**Требования к сборке для GitHub Pages:**
- Если используется Vanilla JS — файлы деплоятся напрямую, без сборщика
- Если используется React/Vite — добавить в `vite.config.js` параметр `base: '/<repo-name>/'` и настроить GitHub Actions для сборки и деплоя в ветку `gh-pages`

Пример GitHub Actions (`/.github/workflows/deploy.yml`):
```yaml
name: Deploy to GitHub Pages
on:
  push:
    branches: [main]
jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: 20
      - run: npm ci
      - run: npm run build
      - uses: peaceiris/actions-gh-pages@v3
        with:
          github_token: ${{ secrets.GITHUB_TOKEN }}
          publish_dir: ./dist
```

Исключение — скрипт парсинга словарей (`scripts/parse_dictionaries.py`): он запускается однократно локально на машине разработчика, результат (`parsed_dictionary.json`) включается в сборку как статический файл.

### 4.2. Стек

Агент выбирает стек самостоятельно исходя из простоты реализации и поддержки. Рекомендуемые варианты:

| Вариант | Когда выбрать |
|---|---|
| Vanilla JS + HTML + CSS | Максимальная простота, нет зависимостей, легко открыть без сборки |
| React + TypeScript + Vite | Если нужен удобный state-менеджмент для сложного UI |

Ограничения: никаких серверных технологий (Node.js, Python-сервер, БД). Все данные — в файлах и `localStorage` браузера.

### 4.5. Парсинг PDF-словарей (ключевой компонент)

**Источник данных** — четыре нормативных словаря на ruslang.ru (Распоряжение № 1102-р от 30.04.2025), распределённые по пяти PDF-файлам (Толковый словарь разбит на две части — А–Н и О–Я):

```
https://ruslang.ru/sites/default/files/doc/normativnyje_slovari/slovar_inostr_slov.pdf
https://ruslang.ru/sites/default/files/doc/normativnyje_slovari/orfograficheskij_slovar.pdf
https://ruslang.ru/sites/default/files/doc/normativnyje_slovari/orfoepicheskij_slovar.pdf
https://ruslang.ru/sites/default/files/doc/normativnyje_slovari/tolkovyj_slovar_chast1_A-N.pdf
https://ruslang.ru/sites/default/files/doc/normativnyje_slovari/tolkovyj_slovar_chast2_O-Ja.pdf
```

**Когда запускается** — однократно локально перед первым деплоем (или при обновлении словарей). Результат `parsed_dictionary.json` коммитится в репозиторий и деплоится как статический файл.

**Скрипт парсинга** (`scripts/parse_dictionaries.py`):

```python
import pdfplumber
import json
import re
from pathlib import Path

DICT_FILES = {
    "foreign_words":  "slovar_inostr_slov.pdf",       # приоритетный для проверки
    "orthographic":   "orfograficheskij_slovar.pdf",
    "orfoepic":       "orfoepicheskij_slovar.pdf",
    "explanatory_1":  "tolkovyj_slovar_chast1_A-N.pdf",
    "explanatory_2":  "tolkovyj_slovar_chast2_O-Ja.pdf",
}

def extract_headwords(pdf_path: str) -> set[str]:
    """
    Извлекает заголовочные слова из словарной статьи.
    Конкретный паттерн уточнить после визуального осмотра PDF.
    """
    words = set()
    with pdfplumber.open(pdf_path) as pdf:
        for page in pdf.pages:
            text = page.extract_text(x_tolerance=2, y_tolerance=2)
            if not text:
                continue
            for line in text.splitlines():
                line = line.strip()
                match = re.match(r'^([А-ЯЁа-яё][а-яёА-ЯЁ\-]{1,40})', line)
                if match:
                    word = match.group(1).lower().strip('-')
                    if len(word) > 2:
                        words.add(word)
    return words

def build_dictionary():
    all_words = {}
    for dict_name, filename in DICT_FILES.items():
        path = Path("pdfs") / filename
        if not path.exists():
            print(f"[SKIP] {filename} не найден — скачайте вручную в папку pdfs/")
            continue
        print(f"[PARSE] {filename}...")
        words = extract_headwords(str(path))
        for word in words:
            if word not in all_words:
                all_words[word] = []
            all_words[word].append(dict_name)
        print(f"  → {len(words)} слов извлечено")

    output = {
        "meta": {
            "source": "Распоряжение Правительства РФ № 1102-р от 30.04.2025",
            "url": "https://ruslang.ru/normativnyje_slovari",
            "total_words": len(all_words)
        },
        "words": all_words  # { "маркетинг": ["orthographic", "foreign_words"], ... }
    }

    with open("data/parsed_dictionary.json", "w", encoding="utf-8") as f:
        json.dump(output, f, ensure_ascii=False, indent=2)
    print(f"\n✓ Готово: {len(all_words)} уникальных слов → data/parsed_dictionary.json")

if __name__ == "__main__":
    build_dictionary()
```

**Важные оговорки:**

1. До запуска скрипта визуально открыть PDF и убедиться, что заголовочные слова выделяются. При необходимости скорректировать `re.match` паттерн.
2. Если PDF окажется сканом — добавить OCR через `pytesseract` с языком `rus`.
3. Точность парсинга ~90–95%. Для MVP приемлемо.
4. Главный словарь — `slovar_inostr_slov.pdf` (~22 000 слов).

**Загрузка словаря в браузере:**

```javascript
// Для Vite + GitHub Pages учитывать base из vite.config.js:
// import.meta.env.BASE_URL + 'data/parsed_dictionary.json'
const response = await fetch('/data/parsed_dictionary.json');
const dictionary = await response.json();

function isWordAllowed(word) {
  return word.toLowerCase() in dictionary.words;
}
```

> Пример использует абсолютный путь. При деплое на GitHub Pages под подпутём репозитория заменить на `import.meta.env.BASE_URL + 'data/parsed_dictionary.json'` (Vite) или настроить `base` в конфиге сборщика.

### 4.6. Типы данных

```typescript
interface CheckResult {
  violations: Violation[];
  stats: {
    total_words: number;
    violations_count: number;
    risk_score: "low" | "medium" | "high";
  };
  corrected_text: string;
}

interface Violation {
  word: string;
  position: { start: number; end: number };
  source: "email_text" | "image_text";  // из какого поля ввода
  type: "LAT_PROHIBITED" | "CYR_NOT_IN_DICT" | "TECH_ABBREV";
  risk: "LOW" | "MEDIUM" | "HIGH";
  norm: string;        // Напр.: "Ч. 6 ст. 3 ФЗ-53"
  norm_url?: string;   // Ссылка на текст нормы
  replacements: string[];
}

interface GlossaryEntry {
  original: string;
  replacements: string[];
  preferred: string;
  type: "LAT_PROHIBITED" | "CYR_NOT_IN_DICT";
}
```

Поле `source` используется в UI: в списке нарушений рядом с каждым показывается бейдж «Текст письма» или «Текст с изображений». Подсветка в textarea применяется только к соответствующему полю. Diff и «применить замены» работают независимо для каждого источника.

### 4.7. Обработка HTML-писем

Если на вход подан HTML-код, перед токенизацией применяется следующая логика очистки:

**Включается в проверку:**
- Текстовое содержимое тегов `<p>`, `<td>`, `<div>`, `<span>`, `<a>`, `<h1>`–`<h6>`, `<li>`, `<button>`, `<label>`
- Значение атрибута `title` у ссылок и кнопок
- Текст внутри `<noscript>`

**Исключается из проверки:**
- Содержимое `alt` у изображений — пользователь сам добавляет текст баннеров в отдельное поле
- Элементы с атрибутом `style="display:none"` или `hidden`
- Элементы с `aria-hidden="true"`
- Preheader-текст (обычно скрытый `<div>` с `max-height:0` или `font-size:0`)
- Содержимое `<style>` и `<script>`
- HTML-комментарии `<!-- -->`

**Метод парсинга:** DOM-парсинг через `DOMParser` браузера, обход дерева с фильтрацией по видимости. Не использовать грубую замену тегов регулярными выражениями — это даёт ложные срабатывания на атрибуты и CSS-классы с латиницей.

### 4.8. Алгоритм токенизации текста

- Разбивать текст по пробелам и пунктуации, сохраняя позиции символов
- Приводить слово к нижнему регистру для поиска по словарю
- Обрабатывать составные слова через дефис как одну единицу
- Игнорировать: email-адреса, URL, числа, HTML-теги (если подан HTML)

### 4.9. Нормализация слов перед поиском по словарю

Нормализация нужна только для кириллических слов (шаги 2–3 алгоритма). Латиница не нормализуется — латинские слова сравниваются точно.

**Проблема:** словарь содержит словоформы («маркетинга», «маркетингу»), а в тексте письма встречаются все падежи и числа. Простой стемминг входного слова с последующим поиском основы в словаре не работает — в словаре лежат словоформы, а не основы.

**Правильное решение — stem-индекс, строится при парсинге PDF:**

Скрипт `parse_dictionaries.py` при извлечении слов дополнительно вычисляет стем каждого слова и строит обратный индекс `stem → [список словоформ]`. Этот индекс сохраняется в `data/parsed_dictionary.json` рядом с `words`.

```python
from snowballstemmer import stemmer as SnowballStemmer

ru_stemmer = SnowballStemmer('russian')

stems_index = {}  # stem → True (достаточно факта наличия)
for word in all_words:
    stem = ru_stemmer.stemWord(word)
    stems_index[stem] = True

output = {
    "meta": { ... },
    "words": all_words,        # точные словоформы из PDF
    "stems": stems_index       # stem-индекс для fallback
}
```

**Двухступенчатый поиск в браузере:**

```javascript
// Шаг 1: точное совпадение в словоформах
if (lower in dictionary.words) return true;

// Шаг 2: fallback — стемминг входного слова, поиск стема в stem-индексе
const stem = russianStemmer(lower);
if (stem in dictionary.stems) return true;

return false; // → CYR_NOT_IN_DICT
```

Оба шага работают в одном пространстве: входное слово стеммируется, стем ищется в заранее построенном `stems`-индексе — а не в `words`. Ложных отрицаний из-за несовпадения форм не будет.

**Если реализация stem-индекса усложняет MVP** — допустимо ограничиться только точным совпадением по `words` и зафиксировать в `README`: «словоформы, отсутствующие в орфографическом словаре, могут давать ложные срабатывания». Stem-индекс добавить в v1.1.

**Важное ограничение:** стемминг (Snowball) ≠ лемматизация. Единичные ошибки возможны в обе стороны. Точная лемматизация (pymorphy3) — при необходимости в v2.0.

## 6. Требования к данным и хранению

### 6.1. Источник нормативных слов — парсинг PDF

Внутренняя база нормативных слов **не ведётся вручную**. Единственный источник — официальные PDF-файлы с ruslang.ru, разобранные скриптом `parse_dictionaries.py` (п. 4.5).

**Порядок подготовки данных при деплое:**

```
1. Скачать PDF в папку pdfs/:
   - slovar_inostr_slov.pdf        ← главный
   - orfograficheskij_slovar.pdf
   - orfoepicheskij_slovar.pdf
   - tolkovyj_slovar_chast1_A-N.pdf
   - tolkovyj_slovar_chast2_O-Ja.pdf

2. Запустить: python scripts/parse_dictionaries.py

3. Полученный parsed_dictionary.json поместить в `data/`
   → сервис готов к работе без перезапуска
```

Ожидаемый размер `parsed_dictionary.json`: 1–3 МБ, 20 000–40 000 слов из всех пяти файлов.

### 6.2. Глоссарий замен (предустановленный)

Глоссарий замен состоит из двух слоёв:

- **`data/glossary.json`** — встроенная база предустановленных пар из п. 3.6. Read-only, не перезаписывается пользователем. Обновляется только при новом деплое.
- **`localStorage` (`user_glossary`)** — пользовательские оверрайды: добавленные или изменённые пары. При поиске замены сначала проверяется `localStorage`, затем `data/glossary.json`. Пользователь видит и редактирует только свои оверрайды через UI.

Такая структура позволяет обновлять встроенный глоссарий через деплой, не затрагивая пользовательские настройки.

### 6.3. История проверок

- Сохранять последние 50 проверок в localStorage (без авторизации)
- При наличии бэкенда — сохранять в БД с возможностью экспорта

### 6.4. Обновление словаря

При выходе обновлённых PDF на ruslang.ru (или при запуске ГИС «Национальный словарный фонд», ожидается к 2027 г.):

1. Заменить PDF-файлы в папке `pdfs/`
2. Перезапустить `parse_dictionaries.py`
3. Заменить `parsed_dictionary.json`

Код сервиса при этом **не изменяется**.

## 7. Ограничения и допущения

1. **Словари доступны в виде PDF, не в виде структурированных данных.** Парсинг производится скриптом `parse_dictionaries.py` (п. 4.5) однократно при деплое. Точность ~90–95% — парсер может пропустить слова с нестандартным форматированием. Для MVP это приемлемо: критичные нарушения (латиница) парсинг не затрагивает.

2. **Наличие русского аналога** — экспертное суждение, не алгоритмически разрешимое полностью. Сервис покрывает известные случаи через глоссарий; сложные случаи передаёт на ручную проверку или в режим проверки через OpenRouter (v2.0).

3. **Корректность перевода при билингвальном использовании** не проверяется. Если пользователь размещает латиницу рядом с русским переводом — это остаётся на его усмотрении. Сервис маркирует латиницу как `LAT_PROHIBITED` и предлагает замену; пользователь сам решает, оставить ли билингвальный вариант. В v2.0 возможна проверка через OpenRouter.

4. **Правоприменительная практика 168-ФЗ только формируется.** Закон действует с 01.03.2026. Сервис реализует наиболее консервативную трактовку.

5. **Равнозначность оформления** (цвет/шрифт/размер по ч. 2 ст. 3 ФЗ-53) в версии 1.0 не проверяется — только текстовое содержимое. Визуальный контроль вынесен в будущую версию.

## 10. Замечания по реализации для AI-агента

1. **Первый шаг — запустить парсер.** До написания кода приложения скачать PDF-файлы в папку `pdfs/` и запустить `parse_dictionaries.py`. Убедиться, что `data/parsed_dictionary.json` содержит >10 000 слов. Если PDF оказался сканом — добавить OCR через `pytesseract` с языком `rus`.

2. **Архитектура данных:** `data/parsed_dictionary.json` — read-only, генерируется скриптом, не редактируется через UI. `data/glossary.json` — встроенная база замен, read-only. Пользовательские оверрайды глоссария — в `localStorage` (`user_glossary`). При поиске замены: сначала `localStorage`, затем файл. Смешивать слои нельзя.

3. **Деплой — GitHub Pages.** Статические файлы (HTML/CSS/JS + JSON-данные) деплоятся через GitHub Actions при пуше в `main`. Для React/Vite — настроить `base` в `vite.config.js` под имя репозитория. Никакого сервера не нужно.

4. **Токенизация** — регулярное выражение с сохранением позиций для корректной подсветки нарушений в исходном тексте.

5. **Юридические ссылки** — хранить в `data/norms.json`, чтобы при изменении практики обновлять без правки логики.

6. **Тестовые данные** — предустановить 5 тестовых текстов с известными нарушениями для демонстрации и QA.

7. **Не изобретать юридические позиции** — все решения о допустимости/недопустимости брать строго из настоящего ТЗ и нормативной базы п. 2.

8. **Норма о запрете иностранных слов** — в коде и комментариях ссылаться на **ч. 6 ст. 3 ФЗ-53** (не ст. 1). Перед финализацией верифицировать по kremlin.ru/acts/bank/22441.
